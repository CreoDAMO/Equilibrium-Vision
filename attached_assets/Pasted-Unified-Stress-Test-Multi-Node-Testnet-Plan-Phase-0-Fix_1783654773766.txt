Phase 2: Adversarial Scenario Suite (arbitrage.stress.test.ts)

Create a dedicated stress test file using Supertest, entirely in‑memory, with full control over block height and pool state.

2.1 Rapid‑Fire Execution

· Loop execute 6 times within one block window.
· Assert the 5th succeeds, 6th is rejected, contract auto‑pauses.
· After unpause, window resets.

2.2 Multi‑Hop Market Crash (Hard Cap)

· Seed a 4‑hop cycle with expected profit 10× arbitrageMaxTradeAmount.
· Assert execute fails completely—no partial swaps, no state changes.
· Rejection occurs before any dex_multi_swap host call.

2.3 Profit Below Minimum

· Construct pool state where Bellman‑Ford finds a negative cycle but net profit is under arbitrage_min_profit.
· Assert rejection.

2.4 Concurrent Execution Attempts (Multi‑Caller) – Event‑Loop Concurrency Stress

· Fire 10 parallel execute calls from different addresses within the same block using Promise.all.
· Because execute_arbitrage spawns variational-ai-cli as a subprocess, the Express event loop yields while waiting for child process stdout—this is exactly where a race condition would manifest if the contract's window counter or execution lock were non‑atomic.
· Assert exactly 5 succeed and the rest are rejected.
· Additional assertion: the execution counter increments exactly once per successful call, even under overlapping subprocess returns.

2.5 Circuit Breaker Reset After Unpause

· After a trip, advance the block beyond the window.
· Assert execute still fails until unpause (admin) is called.
· Verify unpause clears the breaker and resets the window.

2.6 Sliding Window Correctness

· Execute 3 trades in block N, then mine arbitrageWindow + 1 empty blocks.
· Execute 3 more trades; assert the breaker does not trip because old executions have fallen out.

2.7 Governance Parameter Changes Mid‑Test

· Change arbitrageMaxTradeAmount and arbitrageWindow via governance stubs while the contract is running.
· Verify subsequent execute calls respect new values (non‑cached reads).

2.8 Rollback Scenario

· After a reorg that rolls back a block containing an arbitrage execution, verify circuit‑breaker state is correctly restored.
· Leverages the Phase 0 block height fix.

2.9 Partial Success (No Rollback Behavior)

· Simulate a multi‑hop path where leg 1 succeeds but leg 2 slips below the expected profit target.
· Confirm the trade is not rolled back (as documented in LIMITATIONS.md).
· Verify the circuit breaker accounts for the actual profit, not the expected profit, and pool constants remain consistent.

Phase 3: Deterministic Fuzzing with Randomized Market States

Add a fuzzing harness inside the stress test file that:

1. Generates random DEX pools using a seeded PRNG (for reproducibility).
2. Runs the Bellman‑Ford detector (via the Rust CLI) to identify profitable cycles.
3. For each cycle, computes the optimal trade size via the variational solver and submits execute.
4. After each execution, checks invariants:
   · No swap amount exceeds arbitrageMaxTradeAmount.
   · Circuit breaker state is consistent.
   · DEX pool constants remain invariant or strictly non‑decreasing: k' \ge k. Any negative drift, even a single scaled unit, indicates a truncation or rounding vulnerability in the fixed‑point engine.
5. Numerical boundary probing: Dedicated fuzz targets probe the exact edges of both fixed‑point scales—10^{12} for the variational solver (residuals, gradients) and 10^{18} for the blockchain state (residualFp in fork‑choice, DEX amounts). The invariant check uses the same fixed‑point representation the contract uses.

Run thousands of iterations in seconds to gain high confidence that safety rails hold under arbitrary market conditions.
